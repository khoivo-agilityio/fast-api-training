## `uvicorn` venv script has broken shebang in multi-stage Docker build (same root cause as `alembic`)

**Problem:** Even with `UV_PYTHON_PREFERENCE=only-system`, the `uvicorn` venv script
shebang points to `/build/...` (the builder stage path), which doesn't exist in the
runtime stage.

**Symptom:**
```
./docker-entrypoint.sh: line N: /app/.venv/bin/uvicorn: cannot execute: required file not found
```

**Workaround:** Use `python -m uvicorn` in `docker-entrypoint.sh` â€” same fix as
`python -m alembic`. Apply this rule universally: **never call bare venv scripts
inside Docker**; always use `python -m <module>`.

### `ALTER TYPE ADD VALUE` in PostgreSQL cannot be used in the same transaction as DML

**Problem:** PostgreSQL requires `ALTER TYPE â€¦ ADD VALUE` to be fully committed
before any DML (`UPDATE`, `INSERT`) can reference the new enum value. Alembic
wraps all migration steps in a single transaction, so the UPDATE fires before the
ADD VALUE is committed.

**Symptom:**
```
asyncpg.exceptions.UnsafeNewEnumValueUsageError: unsafe use of new value "user"
  of enum type user_role_enum
HINT: New enum values must be committed before they can be used.
```

**Workaround:** In the migration, use `op.get_bind()` to issue raw SQL, explicitly
`COMMIT` after the `ADD VALUE`, then `BEGIN` a new transaction for the DML:
```python
bind = op.get_bind()
bind.execute(sa_text("COMMIT"))
bind.execute(sa_text("ALTER TYPE my_enum ADD VALUE IF NOT EXISTS 'new_val'"))
bind.execute(sa_text("COMMIT"))
bind.execute(sa_text("BEGIN"))
op.execute("UPDATE my_table SET col = 'new_val' WHERE ...")
```

**Date:** 2026-04-14

### Multiple Alembic heads crash container startup

**Problem:** Two migration files with the same parent revision create a branch.
`alembic upgrade head` (singular) fails with:
```
FAILED: Multiple head revisions are present for given argument 'head'
```

**Root cause in this project:** A hand-written migration `a1b2c3d4e5f6` (fake ID,
round timestamp) was committed alongside the autogenerated `a2235df5580e` â€” both
revising `120e14d02e4d`.

**Fix:** Identify the duplicate with `uv run alembic heads`, delete the stale
file, confirm `uv run alembic heads` shows one head. If both are real divergent
branches use `uv run alembic merge heads -m "merge"` instead.

**Date:** 2026-04-14

### `uv` creates venv with broken shebang pointing to `/root/.local/â€¦` in Docker builder stage

**Problem:** When `uv sync` runs in a Docker build stage as root, `uv` defaults to
downloading its own managed Python into `/root/.local/share/uv/python/â€¦`. The venv
scripts (`alembic`, `uvicorn`, etc.) get shebangs pointing to that path. When the
venv is `COPY`-d into the runtime stage, that path no longer exists.

**Symptom:** `./docker-entrypoint.sh: line 5: /app/.venv/bin/alembic: cannot execute: required file not found`

**Workaround:** Two complementary fixes applied:
1. Set `UV_PYTHON_PREFERENCE=only-system` in the `RUN uv sync ...` command to force
   `uv` to use the image's system Python (e.g. `/usr/local/bin/python3`).
2. Use `python -m alembic upgrade head` in the entrypoint instead of bare `alembic`,
   so the script shebang is irrelevant.

**Date:** 2026-04-13

### Base image must be `python:3.14-slim`, not `python:3.12-slim`

**Problem:** `pyproject.toml` specifies `requires-python = ">=3.14"`. Using
`python:3.12-slim` causes `uv sync` to fail with:
`error: No interpreter found for Python >=3.14 in search path`

**Workaround:** Use `python:3.14-slim` as the base image. Despite being a newer
release, the Docker Hub slim image is stable (confirmed working April 2026).

**Correction to plan:** `devops-execution-plan.md` Q4 recommended `python:3.12-slim`
as "pragmatic". This is **wrong** for this project â€” always use `python:3.14-slim`
to match `pyproject.toml`.

### `--chown=appuser:appuser` required on all `COPY` from builder to runtime

**Problem:** Without `--chown`, copied files are owned by root. The `appuser` (uid
1000) cannot execute binaries in the venv, causing `Permission denied` on `python`.

**Workaround:** All `COPY --from=builder` instructions use `--chown=appuser:appuser`.
The `chmod +x docker-entrypoint.sh` line still runs as root (before `USER appuser`
directive), which is correct.

**Date:** 2026-04-13

### Docker BuildKit cache corruption after failed builds

**Problem:** After a failed build that partially wrote layers, subsequent builds may
fail with: `parent snapshot â€¦ does not exist: not found`

**Workaround:** `docker builder prune -f` clears the stale BuildKit cache. Then
`docker build --no-cache â€¦` rebuilds cleanly.

### Tests use aiosqlite (SQLite) but production uses asyncpg (PostgreSQL)

**Problem:** Some PostgreSQL-specific behaviour (e.g. `ON CONFLICT`, array types,
`JSONB`) may not be caught by SQLite-based tests.

**Workaround:** Use `docker-compose.test.yml` (Day 2) to run tests against a real
PostgreSQL instance for integration testing. Keep aiosqlite for fast local unit tests.

### Railway `$PORT` environment variable

**Problem:** Railway injects a `PORT` env var that may differ from the default 8000.
If the app doesn't bind to `$PORT`, Railway's health check fails and the deploy rolls back.

**Workaround:** `Dockerfile` CMD uses `${PORT:-8000}`. `railway.toml` `startCommand`
explicitly references `$PORT`. `Settings` class has `PORT: int = 8000` as fallback.

**Date:** 2026-04-13

### `docker-entrypoint.sh` must be LF line endings (not CRLF)

**Problem:** If the entrypoint script has Windows-style line endings, the container
fails with `/bin/bash^M: bad interpreter`.

**Workaround:** Ensure `.gitattributes` forces LF for `.sh` files:
```
*.sh text eol=lf
```

**Date:** 2026-04-13
